---
title: Dependency Injection
description: Inject services into your resources via constructor injection
---

Resources are resolved from Laravel's service container, which means you can inject any service or dependency into the constructor.

## Basic usage

```php
use BlueBeetle\ApiToolkit\Resources\Resource;
use App\Models\Product;
use App\Services\PricingService;

final class ProductResource extends Resource
{
    protected string $model = Product::class;

    public function __construct(
        private readonly PricingService $pricing,
    ) {
    }

    public function attributes(Product $product): array
    {
        return [
            'name' => $product->name,
            'computed_price' => $this->pricing->calculate($product),
        ];
    }
}
```

## Multiple dependencies

Inject as many services as you need:

```php
final class ProductResource extends Resource
{
    protected string $model = Product::class;

    public function __construct(
        private readonly PricingService $pricing,
        private readonly PermissionService $permissions,
        private readonly InventoryService $inventory,
    ) {
    }

    public function attributes(Product $product): array
    {
        return [
            'name' => $product->name,
            'price' => $this->pricing->calculate($product),
            'in_stock' => $this->inventory->isAvailable($product),
        ];
    }

    public function meta(Product $product): array
    {
        return [
            'can_edit' => $this->permissions->canEdit($product, $this->request?->user()),
            'can_delete' => $this->permissions->canDelete($product, $this->request?->user()),
        ];
    }
}
```

## How it works

Every place that instantiates a resource uses Laravel's `app()` helper instead of `new`:

- `Resource::make()` resolves from the container
- `Response::success()` resolves from the container
- `ResourceCollection` resolves from the container
- `QueryBuilder::fromResource()` resolves from the container
- Relationship resources are resolved from the container

This means DI works consistently everywhere, whether you're serializing a single resource, a collection, or reading query configuration.

## Performance

Resources are resolved from the container once and reused:

- In a **collection**, the resource is resolved once and reused for every item in the collection
- In the **QueryBuilder**, the resource is resolved once and reused across all configuration methods (filters, sorts, includes)
- **Related resources** in relationships are resolved once per relationship type, not per item

Laravel's container resolution is fast. Without constructor dependencies, it's essentially the same as `new`. With dependencies, Laravel caches the reflection metadata.

## Without DI

If your resource has no constructor dependencies, you don't need to define a constructor at all. The container will instantiate it without any arguments:

```php
final class ProductResource extends Resource
{
    protected string $model = Product::class;

    public function attributes(Product $product): array
    {
        return [
            'name' => $product->name,
        ];
    }
}
```
